# Task 1: Prisma schema migration

## Required Skills

Перед выполнением задачи загрузи:
- `/skill:code-writing` — [skills/code-writing/SKILL.md](~/.claude/skills/code-writing/SKILL.md)

## Description

Establish the new data model foundation for the form-builder feature. All subsequent API and UI tasks (Tasks 2–9) depend on this schema being in place before they can start.

The task adds 5 new Prisma models (`FormTemplate`, `FormStep`, `FormField`, `FormSubmission`, `FieldResponse`) and 2 new enums (`FieldType`, `SubmissionStatus`) that power the entire dynamic form-builder system. It also removes the `OnboardingToken` model, which is being replaced by the permanent template URL system. Finally, it extends `prisma/seed.ts` to create a default "Standard Onboarding" template after the admin user is seeded, so the system is immediately usable after deploy.

The `Merchant` model is retained unchanged — it is preserved for future document generation per the user-spec technical decision.

## What to do

1. Add the 5 new models to `prisma/schema.prisma` in the order defined in the tech-spec Data Models section: `FormTemplate`, `FormStep`, `FormField`, `FormSubmission`, `FieldResponse`.
2. Add the 2 new enums: `FieldType` (TEXT, EMAIL, URL, NUMBER, TEXTAREA, FILE, CHECKBOX) and `SubmissionStatus` (NEW, IN_REVIEW, APPROVED, REJECTED).
3. Remove the `OnboardingToken` model from `prisma/schema.prisma`.
4. Run `npx prisma migrate dev --name form_builder_schema` to generate and apply the migration.
5. Verify the migration runs without errors and the Prisma client generates correctly.
6. Extend `prisma/seed.ts`: after the admin user creation block, add a seed block that creates a `FormTemplate` named "Standard Onboarding" with 4 steps mirroring the current hardcoded form (About the Project / Contacts / Business Profile / Documents), including Company Name as a protected field (`isProtected: true`). The seed must be idempotent — check if the template already exists before creating it.
7. Run `npm run seed` and verify the template is created in the DB.

## TDD Anchor

No unit tests for this task — it is pure schema and seed work, not application logic. The correctness of the schema is verified by the migration succeeding and the Prisma client generating TypeScript types that compile cleanly. Downstream tasks (Tasks 2–4) will catch any schema issues when they implement and test API routes against it.

## Details

**Files:**
- `prisma/schema.prisma` — add 5 models + 2 enums, remove `OnboardingToken`
- `prisma/seed.ts` — add default template seeding after admin user block

**Dependencies:** None — this is Wave 1, the foundation for all other tasks.

**Edge cases:**
- Seed idempotency: check `prisma.formTemplate.findFirst({ where: { name: 'Standard Onboarding' } })` before creating; skip with log message if found (matching the existing pattern for user seeding). **Important:** the current `seed.ts` has an early `return` statement after the admin user block — template seeding must be placed BEFORE that return, or the return must be removed/restructured so template seeding always runs regardless of whether the user already exists.
- `@@unique([templateId, order])` on `FormStep` and `@@unique([stepId, order])` on `FormField` — ensure seed assigns correct integer `order` values starting from 0 or 1 (be consistent)
- `FieldResponse.value @db.Text` — required for FILE-type responses that store JSON `{url, mimeType, size}`; do not omit the `@db.Text` annotation
- `FormSubmission.templateName String` — not nullable; snapshot populated at submit time (not in seed)
- `FormSubmission.companyName String` — not nullable; snapshot populated at submit time (not in seed)

**Implementation hints:**
- Follow the exact model definitions from tech-spec Data Models section — field names, types, nullability, and relation decorators must match exactly
- The `fieldKey` field comment in the schema (`// URL-safe key, used in localStorage and value mapping`) should be preserved
- For seed steps, mirror the 4 steps of the current hardcoded form as described in tech-spec: About the Project, Contacts, Business Profile, Documents
- Company Name field: `label: "Company Name"`, `fieldKey: "company-name"`, `type: "TEXT"`, `isProtected: true`, `required: true`, `order: 0` — place in the first step
