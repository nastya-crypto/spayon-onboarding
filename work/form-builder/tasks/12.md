# Task 12: Test Audit

## Required Skills

Before executing this task, load:
- `/skill:test-master` — [skills/test-master/SKILL.md](~/.claude/skills/test-master/SKILL.md)

## Description

Full-feature test quality audit. Reads all test files created for this feature and verifies coverage of the 13 unit test scenarios defined in tech-spec. This is a read-only audit task — no code is written. The goal is to confirm that meaningful, behavior-testing tests exist for every scenario, and to surface any gaps or weak assertions.

The tech-spec defines 13 unit test scenarios (plus localStorage autosave as a 14th scenario) across `buildZodSchema`, API route handlers (templates, submit, submissions, auth), and the `DynamicForm` component. Each must be present and must assert actual behavior, not just that mocks were called.

## What to do

1. Glob for test files across the codebase:
   - `src/**/*.test.ts`
   - `src/**/*.test.tsx`
   - `src/**/*.spec.ts`

2. Read each test file found.

3. Verify coverage of all 13 scenarios from tech-spec Testing Strategy:
   1. `buildZodSchema` — valid inputs per FieldType (TEXT, EMAIL, NUMBER, URL, CHECKBOX)
   2. `buildZodSchema` — invalid inputs (EMAIL rejects non-email, URL rejects non-URL, required field rejects empty)
   3. `buildZodSchema` — optional fields pass with empty/undefined value
   4. Template create happy path — POST `/api/templates` returns 201; response includes Company Name field with `isProtected: true`
   5. Template create validation — POST with no steps returns 400; POST with empty fields array in a step returns 400
   6. Template PATCH removes protected field → 400 "Company Name field cannot be removed"
   7. Submit happy path — POST `/api/templates/[id]/submit` with valid FormData returns 201; `FormSubmission` created with correct `companyName` and `templateName` snapshots; `FieldResponse` records created for every submitted field
   8. Submit template not found → 404
   9. Submit required field missing → 400
   10. Submit rate limit — 6th request from same IP+templateId within window returns 429
   11. Admin auth role check → 403 on all admin routes for non-ADMIN role session
   12. Status transitions valid — PATCH `/api/submissions/[id]/status` with NEW→IN_REVIEW returns 200
   13. Status transitions invalid — unrecognised status value returns 400
   14. localStorage autosave — `DynamicForm` writes `form-draft-[templateId]` on field change; restores on mount; clears on successful submit; silently no-ops when localStorage throws

4. For each scenario, check:
   - Is a test present?
   - Does it assert the correct HTTP status / return value / side effect?
   - Does it verify behavior, not just that mocks were called?
   - Special check for scenario 7: does the submit happy path test verify that `FieldResponse` records were created (not just `FormSubmission`)?

5. Write report to: `work/form-builder/logs/working/task-12/test-audit-report.json`

Report schema:
```json
{
  "summary": {
    "total_scenarios": 14,
    "covered": <number>,
    "missing": <number>,
    "weak": <number>
  },
  "scenarios": [
    {
      "id": 1,
      "name": "buildZodSchema — valid inputs per FieldType",
      "status": "covered | missing | weak",
      "file": "<path or null>",
      "notes": "<any observations>"
    }
  ],
  "overall_verdict": "pass | fail",
  "issues": ["<list of blocking issues if any>"]
}
```

**What counts as "meaningful" assertion:**
- Checks actual response status code or body value
- Checks that a DB record was created/updated (not just that `prisma.create` was called)
- For localStorage: checks that `localStorage.setItem` was called with the correct key and value structure, or that form fields are pre-filled on mount from a mocked localStorage state

**What counts as "weak" (flag but do not auto-fail):**
- Test only checks that a mock was called, with no verification of arguments or side effects
- Test uses `toBeTruthy()` or similar loose assertions instead of specific value checks

**Edge case — scenario 7 (submit happy path):**
This scenario has an explicit extra check: the test must verify `FieldResponse` records were created, not just `FormSubmission`. If the test only asserts the HTTP 201 or only checks `FormSubmission`, mark as `weak`.
