# Task 13: Pre-deploy QA

## Required Skills

Before executing this task, load:
- `/skill:pre-deploy-qa` — [skills/pre-deploy-qa/SKILL.md](~/.claude/skills/pre-deploy-qa/SKILL.md)

## Description

Acceptance testing before deploy. Runs all unit tests, verifies TypeScript compilation, and confirms all acceptance criteria from user-spec and tech-spec.

This is a QA-only task — no code is written. The task verifies that the entire form-builder feature works end-to-end: schema, APIs, admin UI, public form, localStorage autosave, submission flow, and legacy cleanup.

## What to do

1. Run `npm test` → verify all 13 unit test scenarios pass
2. Run `next build` → verify no TypeScript errors
3. Start `npm run dev`, manually verify the 6-step plan from user-spec:
   - **Step 1:** `/dashboard/templates` → New Template → add 2 steps, 2 fields each → Save → template appears in list
   - **Step 2:** Copy Link → paste in new tab → form opens correctly
   - **Step 3:** Fill form → Submit → "Application submitted!" screen shown
   - **Step 4:** Dashboard → find submission → open detail page → all answers visible → change status
   - **Step 5:** Delete template → confirm → open old link → "This link is no longer available"
   - **Step 6:** Fill halfway → close tab → reopen same link → data restored from localStorage
4. Verify all 9 tech-spec acceptance criteria (listed below)
5. Verify all 16 user-spec acceptance criteria (listed below)
6. Document results in `work/form-builder/logs/working/task-13/qa-report.md`

## Details

**Tech-spec acceptance criteria to verify (9 total):**
- `prisma migrate deploy` runs without errors on fresh DB
- All new API routes return correct HTTP status codes (200/201/400/401/403/404/429/500)
- POST `/api/templates/[id]/submit` returns 429 on 6th request from same IP within 1 hour
- DELETE `/api/templates/[id]` returns 200; `FormSubmission` records remain after deletion
- `FormSubmission.companyName` and `templateName` populated correctly at submit time
- `FieldResponse.fieldLabel` snapshot preserved after FormField deletion
- Unit tests pass: field type validation, rate limiting, companyName extraction, localStorage
- No TypeScript compilation errors (`next build` succeeds)
- Existing auth flow (login → dashboard) unchanged

**User-spec acceptance criteria to verify (16 total):**
- "Templates" button on dashboard opens `/dashboard/templates`
- Templates page shows list: name, created date, Edit / Copy Link / Delete buttons
- "New Template" opens editor
- Editor allows setting name, adding steps, and adding fields per step
- Each field configures: type, label, placeholder, required flag
- Fields and steps reorder via ↑ / ↓ buttons
- Cannot save template without at least one step and one field
- "Company Name" field added automatically and cannot be deleted (Delete button hidden)
- "Copy Link" copies `[NEXTAUTH_URL]/onboarding/[templateId]` to clipboard
- Delete shows confirm dialog; after confirmation template removed from list
- Client opens `/onboarding/[templateId]`, sees form by template structure, fills and submits
- After submission, entry appears in dashboard: company name, template name, date, status NEW
- Submission detail page shows all field answers and allows status change
- Opening deleted template link shows "This link is no longer available"
- Form saves draft to localStorage; reopening restores data
- All admin interface text is in English
