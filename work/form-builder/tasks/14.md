## Description

Deploy the form-builder feature to production via Railway auto-deploy. This task covers the final production rollout: creating the required Supabase Storage bucket, pushing to `main` to trigger Railway's Dockerfile-based build, verifying that `prisma migrate deploy` runs cleanly, and confirming the app is healthy end-to-end in production.

All code, tests, and QA are complete (Task 13). This task is purely operational — no code is written.

## What to do

1. Manually create the `form-uploads` bucket in the Supabase dashboard (set as **public** bucket) if it does not already exist.
2. Push current `main` branch to remote — Railway detects the push and auto-deploys via `Dockerfile`.
3. Monitor Railway build logs — confirm `prisma migrate deploy` runs without errors during the build step.
4. Confirm the health check passes: `GET [NEXTAUTH_URL]/api/health` returns HTTP 200.
5. Open the production URL, create a test template, submit a test form, and verify the submission appears in the dashboard.

## Details

**Platform:** Railway — auto-deploys on push to `main` via Dockerfile. Build step runs `prisma generate && next build`. Start command: `npm start`.

**Supabase bucket:** `form-uploads` must be created manually in the Supabase dashboard before or during this deploy. Bucket must be set to **public** (per Decision 6 in tech-spec). Path pattern used by the app: `{submissionId}/{fieldId}-{timestamp}-{sanitizedFilename}`.

**Environment variables** (all must be set in Railway dashboard):
- `DATABASE_URL` — PostgreSQL connection string (Supabase)
- `NEXTAUTH_URL` — Full production URL (e.g. `https://your-app.railway.app`)
- `NEXTAUTH_SECRET` — JWT signing secret
- `SUPABASE_URL` — Supabase project URL
- `SUPABASE_SERVICE_ROLE_KEY` — Service role key (bypasses RLS, server-side only)
- `PORT=3000` — Must be set explicitly on Railway

**Health check:** Railway checks `GET /api/health` within 60 seconds of deploy. On failure: restarts up to 3 times (`ON_FAILURE` restart policy).

**Migration:** `prisma migrate deploy` runs as part of the Docker build step. If the build log shows migration errors (e.g. stale DB state), run `prisma migrate reset --force` locally against the prod DB before retrying — see note in `prisma/seed.ts` header.

**Dependencies:** Task 13 (Pre-deploy QA) must be complete before pushing.
