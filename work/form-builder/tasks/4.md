# Task 4: Submissions API

## Description

Реализовать три admin-only API-маршрута для работы с заявками на онбординг (`FormSubmission`):

- `GET /api/submissions` — список заявок для дашборда
- `GET /api/submissions/[id]` — полная детальная карточка заявки
- `PATCH /api/submissions/[id]/status` — смена статуса с валидацией переходов

Все три маршрута доступны только пользователям с `session.user.role === 'ADMIN'` (Decision 9 в tech-spec). Эти маршруты заменяют функциональность `src/app/api/merchants/route.ts` и `src/app/api/merchants/[id]/status/route.ts` для новой системы шаблонных форм. Маршруты для `merchants` удаляются в Wave 4 (Task 9), поэтому пока не трогаем их.

## What to do

1. Создать `src/app/api/submissions/route.ts`:
   - `GET` — проверить сессию (`getServerSession`) и роль `ADMIN`, вернуть 401/403 при нарушении
   - Запросить `FormSubmission.findMany` с `orderBy: { createdAt: 'desc' }`
   - Вернуть только поля: `id`, `companyName`, `templateName`, `status`, `createdAt` — **email в список не включать**
   - Ответ: `{ submissions: [...] }`

2. Создать `src/app/api/submissions/[id]/route.ts`:
   - `GET` — проверить сессию и роль `ADMIN`
   - Запросить `FormSubmission.findUnique` с `include: { responses: true }`, вернуть 404 если не найдено
   - Вернуть полный объект включая `email` и весь список `FieldResponse` (в том числе orphan-записи, где `fieldId === null`)

3. Создать `src/app/api/submissions/[id]/status/route.ts`:
   - `PATCH` — проверить сессию и роль `ADMIN`
   - Распарсить тело: `{ status: string }`
   - Валидация переходов: допустимы только `NEW→IN_REVIEW`, `IN_REVIEW→APPROVED`, `IN_REVIEW→REJECTED`
   - Найти запись, вернуть 404 если не найдено
   - Проверить текущий статус и желаемый; вернуть 400 при недопустимом переходе
   - Обновить статус, вернуть `{ status: updatedSubmission.status }`

4. Написать unit-тесты (TDD — сначала тесты, потом код):
   - Все три маршрута проверяют роль: без `ADMIN` → 403
   - `GET /api/submissions` — email не включён в ответ
   - `GET /api/submissions/[id]` — orphan responses (fieldId=null) присутствуют в ответе
   - Переходы статусов: `NEW→IN_REVIEW` → 200; `NEW→APPROVED` → 400; невалидный статус → 400

## Details

**Files:**
- `src/app/api/submissions/route.ts` (новый) — GET список заявок (admin)
- `src/app/api/submissions/[id]/route.ts` (новый) — GET деталь заявки (admin)
- `src/app/api/submissions/[id]/status/route.ts` (новый) — PATCH статус заявки (admin)

**Dependencies:**
- Task 1 (Prisma schema) — модели `FormSubmission`, `FieldResponse`, enum `SubmissionStatus` должны быть в схеме
- `src/lib/auth/auth-options.ts` — уже существует, нужен для `getServerSession`

**Edge cases:**
- Orphan FieldResponse (fieldId=null) — включать в ответ детали; admin должен видеть все данные, даже если поле удалено после отправки (Decision — User-Spec Deviations: "Orphan FieldResponse rendering")
- `templateId === null` на FormSubmission (шаблон удалён) — допустимо, возвращаем как есть
- Недопустимый переход статуса: `NEW→APPROVED`, `NEW→REJECTED`, `IN_REVIEW→NEW`, `APPROVED→*`, `REJECTED→*` — все возвращают 400 с осмысленным сообщением
- Попытка повторно поставить тот же статус (напр. `IN_REVIEW→IN_REVIEW`) — трактовать как недопустимый переход → 400

**Implementation hints:**
- Паттерн auth-check — точно как в `merchants/route.ts`: `getServerSession(authOptions)` → если нет сессии `401`, если роль не `ADMIN` → `403`; не использовать `getToken` (паттерн `merchants/[id]/status/route.ts` использует `getToken` — не копировать этот вариант, предпочесть `getServerSession` для единообразия)
- Для валидации переходов создать объект `const VALID_TRANSITIONS: Record<SubmissionStatus, SubmissionStatus[]>` — позволяет легко расширить в будущем
- Поля `select` в Prisma-запросе для list: явно перечислить `{ id, companyName, templateName, status, createdAt }` — не `findMany()` без select, чтобы email точно не попал в ответ
- `SubmissionStatus` enum импортировать из `@/generated/prisma/client`
