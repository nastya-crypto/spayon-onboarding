# Task 7: Submission detail page

## Description

Репурпозим `/dashboard/merchants/[id]` как `/dashboard/submissions/[id]`. Новая страница отображает все данные заявки (FormSubmission) — ответы по полям, сгруппированные по шагам шаблона, — и позволяет менять статус через существующий компонент StatusChanger.

Главная сложность — надёжный рендеринг в edge-cases: удалённый шаблон (templateId=null), удалённые поля (fieldId=null в FieldResponse, так называемые orphan-ответы), и FILE-тип с защитой от XSS через javascript: URI.

Данные загружаются через `GET /api/submissions/[id]`, который возвращает submission вместе с `template.steps[].fields[]` и всеми `responses`. StatusChanger необходимо адаптировать: сейчас он вызывает `/api/merchants/[id]/status`, нужно переключить на `/api/submissions/[id]/status`.

## What to do

1. Создать `src/app/(dashboard)/submissions/[id]/page.tsx` — серверный компонент по образцу `merchants/[id]/page.tsx`:
   - Проверить сессию (`getServerSession`), редирект на `/login` если нет сессии или роль не ADMIN/REVIEWER
   - Fetch через `GET /api/submissions/[id]` (или напрямую через Prisma по образцу существующей страницы — следовать паттерну проекта)
   - Если submission не найден — `notFound()`
   - Отобразить заголовок: companyName как h1, templateName и дата создания, статус-бейдж
   - Если `submission.templateId === null` — показать notice "Template was deleted" (amber/warning стиль, не блокирующий)
   - Рендерить FieldResponse, сгруппированные по шагу: для каждого шага из `template.steps` (если шаблон не удалён) — заголовок шага, затем поля этого шага с их ответами
   - Orphan responses (fieldId=null): вынести в отдельную секцию "Other responses" — рендерить с fieldLabel snapshot
   - FILE-тип: парсить `value` как JSON `{url, mimeType, size}`, валидировать href (`/^https?:\/\//` — только http/https), рендерить как `<a download>` с текстом "Download file" и размером
   - Если URL файла не проходит валидацию — рендерить как plain text без ссылки
   - Передать `submissionId` и `currentStatus` в `<StatusChanger>`

2. Адаптировать `src/components/dashboard/StatusChanger.tsx`:
   - Добавить проп `submissionId: string` (или сделать generic через проп `apiPath: string` — на усмотрение, но предпочтителен явный проп)
   - Переключить endpoint с `/api/merchants/${merchantId}/status` на `/api/submissions/${submissionId}/status`
   - Убрать проп `merchantId` (или сохранить как deprecated alias — убрать чище)
   - Убедиться, что `merchants/[id]/page.tsx` не сломан (он будет удалён в Task 9, но до этого должен компилироваться — при необходимости обновить вызов StatusChanger там тоже)

## Details

**Files:**
- `src/app/(dashboard)/submissions/[id]/page.tsx` (new) — серверный компонент, рендеринг данных submission
- `src/components/dashboard/StatusChanger.tsx` — заменить `merchantId`/`/api/merchants/` на `submissionId`/`/api/submissions/`

**Dependencies:**
- Task 2 (Template management API) — для структуры `template.steps[].fields[]`
- Task 4 (Submissions API) — `GET /api/submissions/[id]` и `PATCH /api/submissions/[id]/status` должны существовать

**Edge cases:**
- `submission.templateId === null` (шаблон удалён): нет `template.steps` для группировки — все responses уходят в секцию "Other responses"; показать notice "Template was deleted"
- `response.fieldId === null` (orphan после удаления поля): использовать `response.fieldLabel` как label; рендерить в секции "Other responses" даже если шаблон жив
- FILE value невалидный JSON: обернуть парсинг в try/catch, показать как plain text если парсинг провалился
- FILE URL не начинается с http:// или https://: рендерить как plain text, НЕ как ссылку (защита от javascript: URI XSS)
- CHECKBOX value: хранится как строка "true"/"false" — рендерить как "Yes"/"No"

**Implementation hints:**
- Для группировки: построить Map `fieldId → response` из `submission.responses`, затем итерировать по шагам и полям шаблона
- Orphan detection: `submission.responses.filter(r => r.fieldId === null)` — плюс responses, чей fieldId не нашёлся ни в одном шаге шаблона
- Валидация FILE URL: `const isValidFileUrl = (url: string) => /^https?:\/\//i.test(url)` — вынести в отдельную функцию для тестирования
- Структура ответа API (Task 4): `{ id, companyName, templateName, templateId, email, status, createdAt, template: { steps: [{ id, title, order, fields: [{ id, label, type, order }] }] } | null, responses: [{ id, fieldId, fieldLabel, value }] }`
- Для `StatusChanger`: самый чистый вариант — заменить проп `merchantId` на `submissionId` и обновить единственный вызов в `merchants/[id]/page.tsx` временно (до удаления в Task 9), либо добавить оба пропа с `merchantId` как опциональный deprecated
