# Task 7: Submission detail page

## Required Skills

Before executing this task, load:
- `/skill:code-writing` — [skills/code-writing/SKILL.md](~/.claude/skills/code-writing/SKILL.md)

## Description

Repurpose `/dashboard/merchants/[id]` as `/dashboard/submissions/[id]`. The new page displays all submission data (FormSubmission) — field responses grouped by template steps — and allows status changes via the existing StatusChanger component.

The main complexity is robust rendering in edge cases: deleted template (templateId=null), deleted fields (fieldId=null in FieldResponse — "orphan responses"), and FILE-type fields with XSS protection against javascript: URIs.

Data is fetched via `GET /api/submissions/[id]`, which returns the submission along with `template.steps[].fields[]` and all `responses`. StatusChanger must be adapted: currently it calls `/api/merchants/[id]/status` and needs to be switched to `/api/submissions/[id]/status`.

## What to do

1. Create `src/app/(dashboard)/submissions/[id]/page.tsx` — server component following the pattern of `merchants/[id]/page.tsx`:
   - Check session (`getServerSession`), redirect to `/login` if no session or role is not ADMIN
   - Fetch via `GET /api/submissions/[id]` (or directly via Prisma following the existing page pattern)
   - If submission not found — `notFound()`
   - Render header: companyName as h1, templateName and creation date, status badge
   - If `submission.templateId === null` — show notice "Template was deleted" (amber/warning style, non-blocking)
   - Render FieldResponse grouped by step: for each step from `template.steps` (if template exists) — step heading, then each field's response
   - Orphan responses (fieldId=null): render in a separate "Other responses" section using fieldLabel snapshot
   - FILE type: parse `value` as JSON `{url, mimeType, size}`, validate href (`/^https?:\/\//i` — http/https only), render as `<a download>` with "Download file" text and size
   - If file URL fails validation — render as plain text without a link
   - Pass `submissionId` and `currentStatus` to `<StatusChanger>`

2. Adapt `src/components/dashboard/StatusChanger.tsx`:
   - Add prop `submissionId: string` (or generic `apiPath: string` — explicit prop preferred)
   - Switch endpoint from `/api/merchants/${merchantId}/status` to `/api/submissions/${submissionId}/status`
   - Remove `merchantId` prop (or keep as optional deprecated alias — removing is cleaner)
   - Ensure `merchants/[id]/page.tsx` still compiles (it will be deleted in Task 9, but must compile until then — update its StatusChanger call if needed)

## Details

**Files:**
- `src/app/(dashboard)/submissions/[id]/page.tsx` (new) — server component rendering submission data
- `src/components/dashboard/StatusChanger.tsx` — replace `merchantId`/`/api/merchants/` with `submissionId`/`/api/submissions/`

**Dependencies:**
- Task 2 (Template management API) — for `template.steps[].fields[]` structure
- Task 4 (Submissions API) — `GET /api/submissions/[id]` and `PATCH /api/submissions/[id]/status` must exist

**Edge cases:**
- `submission.templateId === null` (template deleted): no `template.steps` for grouping — all responses go to "Other responses" section; show "Template was deleted" notice
- `response.fieldId === null` (orphan after field deletion): use `response.fieldLabel` as label; render in "Other responses" even if template is still alive
- FILE value invalid JSON: wrap parsing in try/catch, render as plain text if parsing fails
- FILE URL not starting with http:// or https://: render as plain text, NOT as link (prevents javascript: URI XSS)
- CHECKBOX value: stored as string "true"/"false" — render as "Yes"/"No"

**Implementation hints:**
- For grouping: build Map `fieldId → response` from `submission.responses`, then iterate over template steps and fields
- Orphan detection: `submission.responses.filter(r => r.fieldId === null)` — plus responses whose fieldId is not found in any template step
- FILE URL validation: `const isValidFileUrl = (url: string) => /^https?:\/\//i.test(url)` — extract as separate function for testing
- API response structure (Task 4): `{ id, companyName, templateName, templateId, email, status, createdAt, template: { steps: [{ id, title, order, fields: [{ id, label, type, order }] }] } | null, responses: [{ id, fieldId, fieldLabel, value }] }`
- For `StatusChanger`: cleanest approach — replace `merchantId` prop with `submissionId` and update the one call in `merchants/[id]/page.tsx` temporarily (until deletion in Task 9), or add both props with `merchantId` as optional deprecated
