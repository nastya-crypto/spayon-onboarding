# Task 8: Dynamic public form

## Required Skills

Before executing this task, load:
- `/skill:code-writing` — [skills/code-writing/SKILL.md](~/.claude/skills/code-writing/SKILL.md)

## Description

Build the public-facing form experience for the new template-based onboarding system. Replace the old static `[token]` form with a fully dynamic form rendered from a `FormTemplate` record fetched at runtime.

`page.tsx` is a server component that calls `GET /api/templates/[id]/public`. If the template is not found (404), it renders a "This link is no longer available" page instead of the form. On success it passes the template data to `DynamicForm`.

`DynamicForm` is a client component that iterates over `template.steps[].fields[]` and renders each field according to its `FieldType`. It reuses the existing `ProgressBar` component for step progress. Navigation follows the same Back/Next/Submit pattern as the existing `OnboardingForm` — Back and Next between steps, Submit only on the last step.

Real-time validation runs on every field change (not just on Next): required check for all fields, plus type-specific rules — EMAIL must match email format, URL must be a valid `https://` URL, NUMBER must be finite.

The form header includes an EN/RU language toggle using the same bilingual pattern established in `OnboardingForm.tsx`: a `T` object with `en` and `ru` keys, a `lang` state, and a toggle button showing the inactive language label.

localStorage autosave: on every field change, write the entire field-values map as JSON to key `form-draft-[templateId]`. On mount, read that key and pre-fill the form if data exists. On successful submit, clear the draft. All localStorage access is wrapped in try/catch — errors are silently ignored so the form works even if localStorage is unavailable.

After successful submit the form state is replaced by a success screen: checkmark icon + "Application submitted!" heading. No redirect.

The Submit button is disabled after the first click to prevent double submissions.

## What to do

1. Create `src/app/(onboarding)/onboarding/[templateId]/page.tsx` (server component):
   - Fetch template from `GET /api/templates/[id]/public` using the `templateId` param
   - If response is 404 — render "This link is no longer available" page (centered, simple, consistent with UX guidelines)
   - If response is 200 — render `<DynamicForm template={template} />`
   - Handle unexpected fetch errors gracefully (generic error page or re-throw for Next.js error boundary)

2. Create `src/app/(onboarding)/onboarding/[templateId]/DynamicForm.tsx` (client component):
   - Accept `template` prop typed as the public API response shape (steps with fields)
   - Initialize field-values state as `Record<string, string | boolean>` keyed by `field.fieldKey`
   - On mount: read `localStorage.getItem('form-draft-[templateId]')`, parse JSON, pre-fill state (silently ignore errors)
   - On every field change: write updated values to `localStorage.setItem('form-draft-[templateId]', JSON.stringify(values))` (silently ignore errors)
   - Render `<ProgressBar current={stepIndex} labels={stepTitles[lang]} />` (step titles from template, fallback to step index label if template title has no translation)
   - Render current step's fields using a `renderField(field, value, error)` function that switches on `field.type`
   - Implement `validateStep(stepIndex)`: for each field in the step, check required + type-specific rules; return `Record<string, string>` of errors
   - Back button: go to previous step, clear errors
   - Next button: run `validateStep`, block if errors
   - Submit button (last step only): run `validateStep`, then POST FormData to `/api/templates/[templateId]/submit`; disable button on first click; on success — clear localStorage draft, show success screen; on error — show inline error message, re-enable button
   - EN/RU toggle: same pattern as `OnboardingForm.tsx` — `T` object, `lang` state, toggle button

3. Field rendering by `FieldType`:
   - `TEXT` → `<input type="text">`
   - `EMAIL` → `<input type="email">`
   - `URL` → `<input type="url">`
   - `NUMBER` → `<input type="number">`
   - `TEXTAREA` → `<textarea>`
   - `FILE` → `<input type="file">` (accepts image/jpeg, image/png, application/pdf)
   - `CHECKBOX` → `<input type="checkbox">`
   - Required fields: red asterisk; optional fields: "(optional)" / "(необязательно)" label
   - Error state: red border + inline error text below field (same CSS classes as `OnboardingForm.tsx`)

4. Validation rules per type:
   - All required fields: non-empty value (for checkbox: must be `true`)
   - `EMAIL`: regex `/^[^\s@]+@[^\s@]+\.[^\s@]+$/`
   - `URL`: `new URL(value)` — no throw; also enforce `https?:` prefix
   - `NUMBER`: `!isNaN(Number(value)) && isFinite(Number(value))`
   - `FILE`: file object present (not null)

5. Bilingual strings needed in `T`:
   - `back`, `next`, `submit`, `submitting`
   - `errRequired`, `errEmail`, `errUrl`, `errNumber`
   - `optional`
   - `successTitle` ("Application submitted!" / "Заявка отправлена!")
   - `langBtn` ("RU" / "EN")
   - `unavailableTitle` / `unavailableMsg` for the 404 page (en/ru)

## Details

**Files:**
- `src/app/(onboarding)/onboarding/[templateId]/page.tsx` (new) — server component; fetch template, 404 page or render DynamicForm
- `src/app/(onboarding)/onboarding/[templateId]/DynamicForm.tsx` (new) — client component; all form logic

**Dependencies:**
- Task 3 must be complete (provides `GET /api/templates/[id]/public` and `POST /api/templates/[id]/submit`)
- `src/components/onboarding/ProgressBar.tsx` — existing, no changes needed
- `src/app/(onboarding)/onboarding/[token]/OnboardingForm.tsx` — read-only reference for patterns; do NOT modify

**Edge cases:**
- Template with a single step: Back button disabled on step 0, Submit appears immediately
- Template with no `placeholder` on a field: render input without placeholder attribute (field is nullable)
- FILE field in localStorage: file objects cannot be serialized; skip FILE field values when writing draft (read back as empty); do not break the rest of draft restore
- Submit network error: show error message, re-enable Submit button
- Submit rate-limit (429): show user-friendly message in both EN and RU
- Template fetch returns non-404 error (500): render generic error rather than "no longer available"

**Implementation hints:**
- Reuse the exact same CSS class constants from `OnboardingForm.tsx` for `baseCls`, `okCls`, `errCls` and the `Field` wrapper component — copy them verbatim to keep visual consistency
- For the server fetch in `page.tsx`, use `fetch(\`\${process.env.NEXTAUTH_URL}/api/templates/\${templateId}/public\`, { cache: 'no-store' })` — avoids caching stale template structure
- `ProgressBar` accepts `labels: readonly string[]` — pass step titles for current lang (a derived array, not state)
- For FILE fields in FormData on submit: use `formValues[field.fieldKey]` as a `File` object stored separately from the string values map; handle file state in a `Record<string, File | null>` alongside the string values
- `form-draft-[templateId]` key: substitute actual `templateId` string at runtime, e.g. `\`form-draft-\${template.id}\``
