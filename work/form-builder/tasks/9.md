# Task 9: Remove legacy onboarding system

## Required Skills

Before executing this task, load:
- `/skill:code-writing` — [skills/code-writing/SKILL.md](~/.claude/skills/code-writing/SKILL.md)

## Description

Remove the legacy single-use token onboarding system now that all new functionality is live (Tasks 1–8 complete). The old system consisted of a token-based one-time link flow: admin generated a per-client token via `POST /api/onboarding/create-link`, the client filled the hardcoded 4-step form at `/onboarding/[token]`, and the server stored a `Merchant` record via `POST /api/onboarding/submit`. This entire flow has been superseded by the template-based system introduced in this feature.

Deleting these files eliminates dead code, prevents accidental use of deprecated endpoints (which reference the removed `OnboardingToken` Prisma model), and reduces middleware complexity by removing no-longer-needed allowlist entries.

The `Merchant` model in `prisma/schema.prisma` is **not touched** — it is retained for future document generation per the user-spec technical decision. Only the files listed below are deleted; no schema changes are made in this task.

## What to do

1. Delete the 6 legacy files:
   - `src/app/api/onboarding/create-link/route.ts`
   - `src/app/api/onboarding/submit/route.ts`
   - `src/app/(onboarding)/onboarding/[token]/page.tsx`
   - `src/app/(onboarding)/onboarding/[token]/OnboardingForm.tsx`
   - `src/app/api/merchants/route.ts`
   - `src/app/api/merchants/[id]/status/route.ts`

2. Update `src/middleware.ts`: remove the `pathname.startsWith("/api/onboarding/submit")` entry from the `authorized` callback's allowlist. The `/onboarding` prefix entry must remain — it now covers `/onboarding/[templateId]` used by the new public form.

3. Verify no remaining imports of deleted modules anywhere in the codebase (search for imports referencing `@/app/api/onboarding`, `@/app/(onboarding)/onboarding/[token]`, `@/app/api/merchants`).

4. Run `npm run build` and confirm it passes with no TypeScript errors and no missing-module errors.

## Details

**Files to delete:**
- `src/app/api/onboarding/create-link/route.ts` — legacy token creation endpoint; `prisma.onboardingToken` was removed in Task 1, so this file would cause a build error if left
- `src/app/api/onboarding/submit/route.ts` — legacy form submission handler; still references `prisma.onboardingToken`, `prisma.merchant.create`, `prisma.contact.create`, and Supabase `logos` bucket; entirely replaced by `api/templates/[id]/submit/route.ts`
- `src/app/(onboarding)/onboarding/[token]/page.tsx` — token-based page server component; references `prisma.onboardingToken.findUnique`; replaced by `[templateId]/page.tsx`
- `src/app/(onboarding)/onboarding/[token]/OnboardingForm.tsx` — large client component (~700 lines); hardcoded 4-step form; replaced by `DynamicForm.tsx`
- `src/app/api/merchants/route.ts` — GET list + POST create for `Merchant` records; replaced by `api/submissions/route.ts`
- `src/app/api/merchants/[id]/status/route.ts` — PATCH merchant status; replaced by `api/submissions/[id]/status/route.ts`

**File to modify:**
- `src/middleware.ts` — remove `pathname.startsWith("/api/onboarding/submit")` from the `authorized` callback. The current middleware allowlist contains: `/login`, `/register`, `/onboarding`, `/api/auth`, `/api/onboarding/submit`, `/api/health`. After this task it should contain: `/login`, `/register`, `/onboarding`, `/api/auth`, `/api/health`. Decision 7 from tech-spec already added the new public routes (`/api/templates/:id/public`, `/api/templates/:id/submit`) in Task 3 — verify they are present and do not re-add them here.

**Dependencies:** Tasks 5, 6, 7, 8 must all be complete and merged before this task runs. The new template pages and API routes must be live before the legacy system is removed.

**Edge cases:**
- The `src/app/api/onboarding/` directory may become empty after deleting both files; also delete the now-empty directories `src/app/api/onboarding/create-link/`, `src/app/api/onboarding/submit/`, and `src/app/api/onboarding/` itself if no other files remain
- The `src/app/(onboarding)/onboarding/[token]/` directory should also be removed once both files are deleted
- The `src/app/api/merchants/[id]/` directory should be removed after deleting `status/route.ts` if no other files remain under `[id]/`; likewise `src/app/api/merchants/` itself if empty
- Do NOT delete `src/app/(dashboard)/merchants/` or any merchant-related UI files not in the delete list — check carefully before any deletion
- The `OnboardingForm.tsx` component is only imported by `[token]/page.tsx` (confirmed by reading those files); no other component imports it, so deletion is safe

**Implementation hints:**
- Use `rm -rf` carefully or delete files one by one to avoid accidentally removing more than intended
- After deletions, run `grep -r "OnboardingToken\|onboardingToken\|/api/onboarding\|api/merchants" src/` to confirm no dangling references remain (excluding `prisma/schema.prisma` where `Merchant` legitimately remains)
- The `Merchant` model in the schema is intentionally retained; any grep hits in `prisma/schema.prisma` for `merchant` are expected and correct
